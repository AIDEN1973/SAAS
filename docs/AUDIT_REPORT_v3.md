## 6️⃣ 가장 위험한 구조적 리스크 5가지

### 1. 테스트 커버리지 — 비즈니스 로직 영역 여전히 부족 (🟡 High → 개선 중)

**현황**: 25개 테스트 파일, 약 300+ 테스트 케이스. v2의 17개에서 증가.

**문제**: 928개 소스 파일 대비 2.7% 파일 커버리지. API SDK 유틸리티(validation, cache, rate-limiter)에 집중. 핵심 비즈니스 로직(student CRUD의 full lifecycle, billing 금액 계산, attendance 상태 전이) 테스트가 얇음.

**3년 운영 시**: 리팩토링 시 회귀 버그 감지 실패 → 수동 QA 의존 → 배포 주기 둔화

**완화 요인**: E2E skip이 16→1건으로 대폭 감소. service/industry 레이어에 단위 테스트 파일이 추가됨. 개선 방향은 올바름.

### 2. 단일 업종 검증 — Industry Adapter 미검증 (🟡 Medium-High)

**현황**: academy 1개 업종만 구현. persons→academy_students 패턴은 설계되었으나 salon/gym/school 등 2번째 업종 없음.

**리스크**: 2번째 업종 추가 시 예상치 못한 추상화 누수(leaky abstraction) 발생 가능. 특히 attendance_logs, billing_plans 등 도메인 특화 테이블의 업종 간 공유 가능성 미검증.

**3년 운영 시**: 새 업종 추가에 3~6개월 소요될 수 있음 (설계상 1~2개월 예상이었을 것)

### 3. Edge Function 장애 복구 — 개선 중이나 미완 (🟡 Medium-High)

**현황**: Dead Letter Queue(20260305030000)와 webhook_events 중복 방지가 최근 추가됨. 그러나 `auto-billing-generation`, `financial-automation-batch` 등 금융 Function의 구체적 재시도/보상 트랜잭션 체계 미확인.

**리스크**: 자동 청구 실패 시 매출 누락. 단일 실패가 cascade되지 않는 것은 확인이 필요.

**완화 요인**: DLQ 인프라가 추가된 것은 올바른 방향.

### 4. 대형 페이지 컴포넌트 비대화 (🟡 Medium)

**현황**: AttendancePage.tsx 2,185줄, ClassesPage.tsx 2,039줄, HomePage.tsx 1,425줄, App.tsx 841줄

**리스크**: 단일 파일 2,000줄 이상은 인지 부하(cognitive load) 임계점 초과. 새 개발자 온보딩 시 이해 시간 증가. 버그 발생 시 원인 추적 어려움.

**완화 요인**: React Query + 커스텀 훅으로 데이터 로직은 분리되어 있어, 실제 "비즈니스 로직"보다는 "UI 조합"이 비대한 것. 리팩토링 가능.

### 5. 운영 모니터링 — 기본 수준 (🟡 Medium)

**현황**: error_logs, edge_function_logs, frontend_error_logs 테이블 존재. Sentry 통합 설정. 그러나 구체적인 알람 정책(P50/P99 응답시간, 에러율 threshold, 자동 스케일링)이 코드에서 보이지 않음.

**리스크**: 프로덕션 장애 발생 시 인지/대응 시간이 길어질 수 있음. B2B SaaS에서 SLA 위반은 계약 위반.

**3년 운영 시**: 고객 수 증가 → 장애 영향 범위 확대 → 모니터링 없이는 "고객이 먼저 발견"하는 상황

- **51개 Edge Functions** — AI Briefing, ChatOps, 자동 청구, 자동 출결까지. 스텁이 아닌 실구현
- **Schema-Driven UI Engine** — 동적 폼 생성, 버전 관리, Semantic Diff. SaaS 플랫폼의 핵심 차별화
- **문서화 40개 + CLAUDE.md** — AI 에이전트 협업까지 설계에 포함. 미래지향적
- **감사 피드백 반영 속도** — v2→v3 사이에 품질 지표 70~93% 개선. 실행력 증명
