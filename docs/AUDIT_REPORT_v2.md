### 타입 안전성 실태

ESLint에서 `@typescript-eslint/no-explicit-any: "error"`를 선언했지만, 실제로:

- `packages/schema-engine` — SchemaField.tsx 혼자 `as any` 20건. 동적 스키마 처리 시 타입 포기
- `packages/ui-core` — Input.tsx(10건), Textarea.tsx(8건), ChatOpsPanel(5건) 등 핵심 컴포넌트에서 `as any` 남용
- `infra/supabase/functions` — Edge Function 공유 모듈에서 타입 우회 다수

**판정: "any 금지" 정책은 존재하나, 현실에서는 상당히 타협됨. 규칙과 현실의 괴리.**

### 테스트 실태 (가장 큰 교정 항목)

**실제 존재하는 테스트:**

| 카테고리 | 파일 | 테스트 수 | 상태 |
|----------|------|-----------|------|
| API SDK 단위 | cache, rate-limiter, validation | ~145건 | **활성** |
| Hook 단위 | useStudent, useAuth, useBilling | ~48건 | **활성** |
| E2E (root) | login, student, attendance, billing, rbac | ~33건 | **16건 skip** |

**총 약 335개 test/it 블록, 실질 활성 약 300개, skip 약 35개**

v1에서 "249개 테스트 파일"이라 보고한 것은 **오류**. 실제 테스트 파일은 17개이며, 그 안에 약 300개의 활성 케이스가 존재. 그러나 908개 소스 파일 대비 17개 테스트 파일은 **커버리지 극히 낮음**. 특히 E2E의 핵심 시나리오(학생 등록, 출결 체크, 청구) 대부분이 `.skip` 상태.

## 6️⃣ 가장 위험한 구조적 리스크 5가지

### 1. 테스트 커버리지 — 비즈니스 로직 영역 공백 (🔴 Critical)

- **v1에서 과소평가됨.** 테스트 300개 중 대부분이 validation/cache/rate-limiter 유틸리티에 집중
- **student CRUD, attendance, billing, teacher** — 핵심 도메인의 서비스/인더스트리 레이어 테스트 **0건**
- E2E 핵심 시나리오 16개 `.skip` — "학생 등록", "출결 체크", "청구서 조회", "RBAC" 모두 비활성
- **3년 운영 시**: 리팩토링/기능 추가마다 회귀 버그 발생 → 수동 QA 비용 급증 → 개발 속도 50% 이하로 저하

### 2. 타입 안전성 구멍 (🔴 High)

- ESLint에 `no-explicit-any: "error"` 선언했으나, `eslint-disable` 101건으로 우회
- Schema Engine, UI Core 핵심 컴포넌트에서 `as any` 집중 사용
- **리스크**: 런타임 타입 에러가 프로덕션에서 발생할 가능성. 특히 Schema Engine의 동적 폼 렌더링에서 예측 불가능한 크래시 가능

### 3. TODO 181건 — 미완성 기능 부채 (🟡 High)

- `industry-academy/service.ts`의 `TODO: class_name 처리` — 학생에 반 이름이 매핑 안 되는 상태
- E2E 테스트 skip 사유: "DB 초기화 필요", "/students 페이지 구현 후 활성화" — 인프라 준비 미완
- **리스크**: 사용자 관점에서 "안 되는 기능"이 곳곳에 숨어있을 가능성

### 4. Edge Function 장애 복구 부재 (🟡 High)

- `auto-billing-generation`, `financial-automation-batch` 등 금융 Function에 재시도/DLQ 없음
- **리스크**: 자동 청구 실패 시 매출 누락, 고객 이탈

### 5. `limit: 5000` 클라이언트 전체 로드 (🟡 Medium-High)

- `fetchStudents`에서 `limit: 5000`으로 전체 데이터 로드
- 서버 페이지네이션(`useStudentsPaged`) 존재하나, 일부 경로에서 여전히 전체 로드 사용
- **리스크**: 대형 학원(5,000명+)에서 성능 저하

## 7️⃣ 최종 총평

### 냉정한 업계 포지션

**"아키텍처 설계는 시리즈 A급, 품질 보증은 해커톤급"**

이 프로젝트는 두 개의 극단이 공존한다:

**강점 (상위 10% 수준):**

- 6계층 Clean Architecture + ESLint 555줄 강제 — 국내 SaaS 중 이 수준의 아키텍처 거버넌스를 가진 프로젝트는 5% 미만
- 49개 Edge Function — 자동화/AI 파이프라인이 실제 구현되어 있음 (스텁이 아님)
- UI Core 149개 컴포넌트 — 실제 프로덕션 수준 구현 (Button 231줄, Input 384줄, DataTable 1000줄+)
- 문서화 24개 핵심 문서 + CLAUDE.md — AI 에이전트와의 협업까지 고려한 설계

**약점 (하위 30% 수준):**

- 비즈니스 로직 테스트 사실상 0건 — 유틸리티 테스트만으로는 제품 품질을 보증할 수 없음
- `any` 104건 + `as any` 81건 — "TypeScript strict mode"라고 말하지만 현실은 구멍 다수
- TODO 181건 — 완성되지 않은 기능이 코드 곳곳에 산재
- 핵심 E2E 시나리오 대부분 `.skip` — "테스트가 있다"와 "테스트가 통과한다"는 다른 이야기

### 사업적으로 의미 있는 수준인가?

**예, 그러나 조건부.**

기능적으로는 상용 서비스로 출시 가능하다. 그러나 현재 상태에서 "과금"을 시작하면:

- billing 관련 E2E가 `.skip`인 상태에서 자동 청구 오류 발생 시 → 법적 리스크
- RBAC 테스트 5건 모두 `.skip` → 권한 이슈로 데이터 유출 가능성
- Schema Engine의 `as any` 20건 → 커스텀 필드 사용 시 예측 불가 크래시

**"기술 데모"로서는 훌륭, "과금 서비스"로서는 아직 불안정.**

### 상위 10% SaaS로 갈 수 있는가?

**갈 수 있다. 아키텍처 기반이 이미 상위 10%에 있기 때문.**

그러나 필요한 것은 구체적으로:

1. **테스트 복구** (2~3주): `.skip` 16건 활성화 + 비즈니스 로직 단위 테스트 최소 50건 추가. 특히 `academyService.getStudents`, `createStudent`, `attendance` 로직
2. **타입 정리** (1~2주): `as any` 81건 중 Schema Engine(20건) + UI Core(23건) 집중 제거. 제네릭 또는 `unknown` + type guard로 전환
